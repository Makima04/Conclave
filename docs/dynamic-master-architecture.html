Conclave 动态 Master Agent 架构

多智能体角色扮演 & 写作平台技术架构文档

1. 概述

Conclave 采用 4 层流水线 驱动用户交互：理解层 → 决策层 → 执行层 → 合成层。核心思想是通过一个 Master Agent（总控智能体）动态编排多个子 Agent，按需创建、激活和回收角色，而非将所有信息塞进一个巨大的 prompt。

2. 架构总览

用户输入
  ↓
+-------------------------------+
| 解析层 (Parser Agent)        |  ← 意图识别 + 实体提取
+-------------------------------+
  ↓
+-------------------------------+
| 决策层 (Master Agent)        |  ← 动态规划 + Agent 调度
+-------------------------------+
  ↓
+-------------------------------+
| 执行层                        |
|  NPC Agent  User Agent       |  ← 独立执行、互不干扰
+-------------------------------+

+-------------------------------+
| 合成层                        |
|  Director + Writer + State   |  ← 叙事整合 + 状态更新
+-------------------------------+
  ↓
输出给用户

3. 四层详解

Layer 1 -- 理解层 (Parser Agent)

职责：将用户的自然语言输入解析为结构化 JSON，供 Master Agent 消费。

  • 意图类型 (intent)：互动 / 剧情安排 / 系统指令 / 剧情推进
  • 动作类型 (action)：对话 / 攻击 / 探索 / 使用物品 / 新角色 / 数值修改
// Parser 输出示例
{
  "intent": "互动",
  "action": "对话",
  "targets": ["npc_艾莉丝"],
  "content": "艾莉丝，你昨天说的那件事...",
  "raw": "我跟艾莉丝说：你昨天说的那件事..."
}

Layer 2 -- 决策层 (Master Agent)

职责：根据 Parser 输出制定执行计划，决定调用哪些子 Agent 以及它们的参数。

// Master 执行计划示例
{
  "plan": [
    { "agent": "npc_艾莉丝", "action": "respond", "inject": {} },
    { "agent": "director",    "action": "evaluate_pacing" },
    { "agent": "writer",      "action": "compose" }
  ],
  "new_agents": [],
  "cool_agents": []
}

上下文管理策略：Master 只接收各子 Agent 的状态摘要（如角色情绪、最近行动），而非完整输出。这确保 Master 的 context window 保持精简。

Layer 3 -- 执行层

职责：各子 Agent 独立执行 Master 分配的任务，互不直接通信。

  • NPC Agent：依据世界观 + 角色档案生成角色反应
  • User Agent：（可选）模拟用户角色的行动

Turn State：共享数据存储区，记录当前轮次的公共信息（场景状态、临时变量等）。

Inject 机制：Master 通过 inject 字段向子 Agent 注入额外上下文（如「你刚刚看到艾莉丝哭了」），实现跨 Agent 信息传递。

Layer 4 -- 合成层

  • Director Agent（节奏控制）：评估当前叙事节奏，决定是否需要加快/放慢、切换场景或引入新冲突
  • Writer Agent（文本生成）：整合所有子 Agent 输出，生成连贯的叙事文本，保持统一文风
  • State Agent（状态更新）：解析输出中的状态变化，更新数据库（角色关系、物品、数值等）

4. 子 Agent 清单

Agent类型上下文生命周期
Master总控各子 Agent 状态摘要永久
Parser理解解析规则永久
npc_{角色名}执行世界观 + 角色档案动态
User Agent执行用户角色设定按需激活
Director合成叙事结构知识永久
Writer合成文风预设永久
State合成状态解析规则永久

5. 生命周期管理

动态 Agent（如 npc_*）遵循以下状态流转：

🌱 创建 (active) → ✅ 活跃 → 💧 冷却 → 💀 退休 / 死亡

冷却规则

触发条件说明
轮数冷却N 轮未被调用（默认 10）
事件触发Master 主动冷却某 Agent
角色退场进入退休状态，保留历史但不再调用
角色死亡永久归档，上下文冻结

6. 场景走查

场景 1：用户与 NPC 对话
  1. 解析层：Parser 识别 intent=互动，action=对话，target=npc_艾莉丝
  2. 决策层：Master 检查艾莉丝 Agent 是否活跃 → 若在冷却则重新激活 → 生成调用计划
  3. 执行层：npc_艾莉丝 根据角色档案和当前剧情生成回复
  4. 合成层：Director 评估节奏 → Writer 整合为叙事文本 → State 更新对话历史
场景 2：战斗场景（User Agent）
  1. 解析层：intent=互动，action=攻击，targets=[敌人列表]
  2. 决策层：Master 激活 User Agent + 相关 NPC Agent（敌人、队友）
  3. 执行层：User Agent 生成用户行动 → NPC Agent 各自反应（通过 inject 获取战斗状态）
  4. 合成层：Director 控制战斗节奏 → Writer 生成战斗叙事 → State 扣减 HP / 更新状态
场景 3：新角色登场
  1. 解析层：intent=剧情安排，action=新角色，携带角色设定
  2. 决策层：Master 创建新 npc_{角色名} Agent，注入世界观 + 角色档案
  3. 执行层：新 Agent 生成登场行为
  4. 合成层：Writer 将登场融入叙事 → State 注册新角色到数据库
场景 4：纯剧情推进
  1. 解析层：intent=剧情推进，action 依赖具体描述
  2. 决策层：Master 可能不调用任何 NPC，只激活 Director + Writer
  3. 执行层：无执行层活动（或仅 User Agent 旁白）
  4. 合成层：Director 判断时间/场景推进 → Writer 生成过渡叙事
场景 5：数值变化
  1. 解析层：intent=系统指令，action=数值修改，携带具体参数
  2. 决策层：Master 判断是否需要叙事包裹（修改数值是否需要剧情解释）
  3. 执行层：通常无执行层参与
  4. 合成层：State 直接更新数据库；Writer 可选择性生成数值变化的叙事描述

7. 信息流控制

🔎 总控上下文管理

Master Agent 只接收摘要，不接收子 Agent 的完整输出。每个子 Agent 执行完毕后，生成一份结构化摘要（角色情绪、关键行动、状态变化），Master 依据这些摘要做下一步决策。

这样做的好处：Master 的 context 不会因子 Agent 输出增长而膨胀，保持调度效率。

🔗 子 Agent 间传递

子 Agent 之间不直接通信。跨 Agent 信息传递完全由 Master 通过 inject 字段控制：

{
  "agent": "npc_艾莉丝",
  "inject": {
    "recent_event": "玩家刚才偷走了你的钥匙",
    "mood_hint": "警惕"
  }
}

这确保信息流动可控、可审计，避免子 Agent 之间的幻觉叠加。

8. 用户可配置参数

参数默认值说明
master_model用户选择总控 Agent 使用的 LLM 模型
sub_agent_model用户选择子 Agent 使用的 LLM 模型（可低于 Master 以节省成本）
cooldown_turns10Agent 未被调用的轮数阈值，超过后进入冷却
user_auto_modeaskUser Agent 自动模式：ask（每次询问）/ auto（自动执行）
max_active_agents8最大同时活跃 Agent 数量，超出时强制冷却最久未用 Agent
parser_enabledtrue是否启用解析 Agent（关闭则 Master 直接处理原始输入）

9. 世界书集成

世界书 (World Book) 是用户编写的设定集合，系统按以下流程将其注入 Agent 上下文：

世界书文件 → 解析 & 分类 → 按角色分配 → 注入 Agent 上下文
  • 全局设定（世界观、魔法体系等）→ 注入所有 NPC Agent
  • 角色档案（性格、背景、关系）→ 注入对应 NPC Agent
  • 场景设定（地点描述、氛围）→ 注入 Turn State 供所有 Agent 读取
  • 叙事规则（文风、禁忌）→ 注入 Writer Agent
