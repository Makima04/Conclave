排序原则

先定义契约，再写实现

跨后端、前端、Runtime、记忆和内容包共享的数据结构必须先有 schema 或规范。

先跑通单轮，再扩展多 Agent

首个可用版本先支持 single_agent 最小闭环，再加入 Director、WorldJudge、Consistency 和高级图。

先保证边界，再开放扩展

插件、自由图、自定义 artifact 必须在权限、沙箱、预算、trace 和 proposal + commit 稳定后再开放。

Markdown 与 HTML 同步

每份正式文档必须同时维护 Markdown 和 HTML 版本，入口、阅读顺序、待补列表和实现进度也要同步。

P0：实现前必补

P0 是进入代码实现前的前置工作。它们不直接产出产品界面，但决定后续实现是否能保持一致。

P1：最小可运行 MVP

P1 目标是做出一个可自托管、能聊天、能流式输出、能记录 trace、能保存基础记忆的最小系统。

后端骨架

Rust API

使用 axum + tokio + sqlx + SQLite，实现健康检查、会话、消息、流式输出和 provider adapter 边界。

前端骨架

React UI

实现会话列表、聊天 / 写作界面、流式输出展示、角色卡和世界书编辑入口。

Runtime

Single Agent

实现 UserInput -> Npc/Writer -> Memory，每轮记录 trace 和记忆候选。

基础记忆：Recent Context、Scene Summary、Structured State、Event Ledger、Foreshadowing Registry。

P1 完成标准：

  • 用户能创建会话并发送消息。
  • 输出支持流式展示。
  • 每轮有可查看 trace。
  • 基础状态、事件和伏笔能写入并在后续轮次召回。

P1 骨架已完成（2026-06-04）。 后端 Rust + axum + SQLite 骨架、前端 React + Vite 骨架、Single Agent Runtime、OpenAI 兼容 Provider Adapter、5 层记忆系统和 API 端点已实现。可运行 cargo run（后端）和 npm run dev（前端）启动。

P2：平台核心差异化

P2 开始体现本项目不是普通聊天 UI，而是受控多 Agent RP / 写作 Runtime。

动态总控多 Agent 架构

已废弃旧固定模板图方案，实现 Dynamic Master 架构：

  • 4 层流水线：Parser → Master → Sub-agents → Writer → Compression
  • Parser Agent：用户输入 → 结构化 ParsedIntent
  • Master Agent：基于上下文 + 意图 + 子 Agent 摘要，动态生成 MasterPlan
  • 子 Agent 执行：独立 LLM 调用、上下文隔离、输出写入 Turn State
  • Compression Agent：后处理压缩，生成 scene_summary + events + structured_events + foreshadowing + state_changes

Agent 生命周期管理

  • Master 动态创建、冷却、退休、删除子 Agent
  • 连续 N 轮不活跃自动冷却（用户可配置，默认 10）
  • 用户可通过 Agent 管理面板手动干预

结构化事件召回

  • structured_events 表存储压缩 Agent 生成的结构化事件
  • recall.rs 实现关键词召回（预留向量召回）
  • 双层去重：精确哈希 + 3 轮窗口语义去重

Proposal + Commit

Memory、Tool、Plugin、Agent 都只能提出候选变更。Runtime 负责校验 schema、权限、风险等级和冲突，再提交到主状态。

ContextBundle 权限隔离

每个 Agent 只拿自己有权读取的信息：

  • NPC 拿 scene + events + foreshadowing + recalled_structured_events
  • Writer 拿 scene + foreshadowing + 本轮 Agent 输出
  • Director 拿完整概览
  • User Proxy 拿 scene + 最近 events
  • NPC 不能读取导演计划或其他 NPC 内心

消息管理

  • 消息编辑、删除、重新生成
  • 变体（variants）系统：重新生成时保留历史版本
  • 自动标题生成（首轮后 LLM 生成会话标题）

P2 完成标准：

  • multi_agent 模式下 4 层流水线完整运行。
  • Master 能根据用户输入动态调度子 Agent。
  • 子 Agent 上下文隔离正确（NPC 不含导演计划）。
  • 压缩 Agent 每轮生成结构化状态并持久化。
  • 结构化事件可被正确召回。
  • trace 能解释本轮为什么召唤某个 Agent、采用哪些记忆。
  • 用户可通过 Agent 管理面板查看和操作所有 Agent。

P2 核心功能已实现（2026-06-05）。 动态总控架构（Parser → Master → Sub-agents → Writer → Compression）、Agent 生命周期管理、结构化事件召回、上下文隔离、消息管理和 Agent 管理 UI 已完成。Consistency 检查（OOC 检测、世界事实冲突等）留待 P3。

P3：高级创作者能力

Agent Graph 编辑

使用 React Flow 做图编辑，支持节点、边、条件、模型、预算和权限配置。导入图包必须经过 schema 校验和安全限制。

Artifact Renderer

聊天消息只引用 artifact_id 和 version，高级 UI 才允许 iframe sandbox 自定义代码。离屏 artifact 卸载或转静态快照。

SillyTavern 迁移

支持导入角色卡、世界书和预设，把历史上下文迁移为摘要、事件账本和结构化状态候选。对无法可靠迁移的隐藏设定和旧记忆显示风险提示。

P3 完成标准：

  • 高级用户能编辑和导入受限 Agent Graph Pack。
  • Artifact 不污染主页面，也不会随长会话无限增加 DOM。
  • SillyTavern 内容能进入平台内容包模型。

P4：插件与生态

插件系统

  • 插件 manifest。
  • 权限申请与用户授权。
  • 代码节点沙箱。
  • 工具调用预算和超时。
  • 插件输出结构化校验。

图包与插件分发

支持 TrustedGraphPack / UnverifiedGraphPack 分级、版本兼容声明、风险提示、导入报告、示例包和测试夹具。

P4 完成标准：

  • 插件不能默认读取隐藏世界书、网络、本地文件或长期状态。
  • 插件只能提交候选变更，不能直接改主账本。
  • 第三方图包质量风险和平台安全责任边界清晰展示。
